## 1. 目的
Spring Boot で、=LOVE ファンサイト向けの参照系 API を実装する。  
ただし設計は将来的なマルチサイト展開、複数グループ対応、履歴管理、検索拡張を前提とし、`site` 起点でデータを扱えるようにする。

---

## 2. バックエンド実装スコープ

### 2.1 今回実装する API
- サイト情報取得
- 楽曲一覧取得
- 楽曲詳細取得
- 楽曲コール取得
- メンバー一覧取得
- メンバー詳細取得
- 作品一覧取得
- 検索条件用マスタ取得

### 2.2 今回実装しない API
- 管理画面向け CRUD
- 認証
- 画像アップロード
- サイト新規作成
- サイト更新
- サイト公開 / 非公開切替
- 利用者ごとのテーマ編集

---

## 3. 実装方針

## 3.1 パッケージ構成案
```text
com.example.fansite
├─ common
│  ├─ exception
│  ├─ response
│  ├─ config
│  └─ util
├─ domain
│  ├─ site
│  ├─ group
│  ├─ member
│  ├─ release
│  ├─ song
│  ├─ alias
│  ├─ call
│  └─ master
├─ application
│  ├─ site
│  ├─ member
│  ├─ release
│  ├─ song
│  ├─ search
│  └─ call
├─ infrastructure
│  ├─ persistence
│  └─ external
└─ presentation
   └─ api
```

## 3.2 レイヤ方針
- `presentation`: Controller / request / response DTO
- `application`: UseCase / Service
- `domain`: Entity / domain service / repository interface
- `infrastructure`: JPA 実装、DBアクセス

## 3.3 拡張性のための重要ルール
- 楽曲と作品は `release_song` で関連付ける
- メンバー所属は `member_group_history` を正とする
- 検索用の別名は `song_alias`, `member_alias` へ分離する
- コール種別は `call_type_master` を参照し、フロントに表示メタデータも返せるようにする
- `site` は将来のユーザー作成サービスに備えて、所有者と公開状態を保持できるようにする
- API DTO は「現在画面で使う最小項目」と「将来画面で再利用しやすい関連情報」を両立させる

## 4. ステップ別実装

## Step 1. 共通基盤整備
### 実施内容
- 例外ハンドリング共通化

- 正常系 / 異常系レスポンス形式が揃う
- API バージョンと requestId の出力方針が固まる

## Step 2. DB スキーマ実装
### 実施内容
- `site`
- `site_theme_setting`
- `idol_group`
- `member`
- `member_group_history`
- `single_release`
- `song`
- `release_song`
- `song_member`
- `song_alias`
- `member_alias`
- `call_type_master`
- `song_call_block`
- `song_call_line`
- `song_call_item`

### 完了条件
- ローカル DB でマイグレーション可能
- 初期データ投入が可能
- 将来の多作品収録、所属履歴、別名追加で再設計不要

### 補足
- Flyway または Liquibase の採用を推奨
- 本番向けに DDL をコード管理すること
- 参照系の主要クエリを想定した複合インデックスを先に定義すること
- 認証基盤は今回未実装のため、所有者参照列は一旦 nullable とし、後日ユーザー管理と連携できるようにする
- 公開制御は今回未実装のため、参照系 API は `PUBLISHED` 相当のサイトを返す前提に寄せておく

---

## Step 3. マスタ / 検索基盤実装
### 実施内容
- `call_type_master` の参照 API または内部参照基盤作成
- 楽曲検索用に `song_alias` を検索対象へ含める
- メンバー検索用に `member_alias` を検索対象へ含める
- 並び順とデフォルトソートの共通ルール定義

### 完了条件
- 楽曲名以外の別名検索を後方互換で追加できる
- コール種別の追加で API 破壊的変更が不要

---

## Step 4. 楽曲一覧 API 実装
### API
- `GET /api/v1/sites/{siteKey}/songs`

### 必須要件
- 楽曲名検索
- 別名検索
- 作品絞り込み
- 表題曲フラグ絞り込み
- オリジナル歌唱メンバー取得
- 代表作品取得
- ソート対応

### 完了条件
- 一覧画面が必要とする情報を 1 API で取得できる
- N+1 問題に配慮している
- `release_song` 経由でも一覧性能が大きく落ちない

### 実装メモ
- 一覧はカード描画しやすい DTO に最適化する
- 作品は `primaryRelease` を返しつつ、将来用に `releases` 追加可能な DTO 設計にする
- 別名検索は本体テーブルへ列追加せず関連テーブルで吸収する

---

## Step 5. 楽曲詳細 API 実装
### API
- `GET /api/v1/sites/{siteKey}/songs/{songId}`

### 必須要件
- 楽曲基本情報
- 代表作品情報
- 収録作品一覧
- 表題曲か否か
- オリジナル歌唱メンバー
- 説明文
- コール有無

### 完了条件
- 楽曲詳細ヘッダーに必要な情報が返る
- 将来の作品追加表示でも API 再設計が不要

---

## Step 6. 楽曲コール API 実装
### API
- `GET /api/v1/sites/{siteKey}/songs/{songId}/calls`

### 必須要件
- 歌詞ブロックごとに返せる
- 歌詞行とコール行の対応が取れる
- コール種別が識別できる
- 表示順を保証する
- `callTypeCode` に加えて表示ラベル・色・アイコンキーを返せる

### 返却イメージ
```json
{
  "songId": 101,
  "blocks": [
    {
      "blockType": "VERSE",
      "orderNo": 1,
      "lines": [
        {
          "lineNo": 1,
          "lyrics": "君に会えたその日から",
          "calls": [
            {
              "callTypeCode": "CHANT",
              "callTypeLabel": "掛け声",
              "text": "はい！",
              "style": {
                "colorHex": "#F06292",
                "iconKey": "mic"
              }
            }
          ]
        }
      ]
    }
  ]
}
```

### 完了条件
- フロントが「歌詞の下にコール」を素直に表示できる
- 種別追加時に API の型変更が最小限で済む

---

## Step 7. メンバー API 実装
### API
- `GET /api/v1/sites/{siteKey}/members`
- `GET /api/v1/sites/{siteKey}/members/{memberId}`

### 必須要件

- 詳細で誕生日、メンバーカラー、出身地、ひとことを返す
- 現在所属グループを返す
- 将来用に所属履歴を返せる構造にする

### 完了条件
- メンバー一覧 / 詳細ページの表示に必要十分なデータが取れる
- 卒業、再加入、兼任に対応できる基盤ができている

---

## Step 8. 作品 API 実装

### 完了条件

- 楽曲初期データ

## 5. API 共通仕様

## 5.1 ステータスコード
- 200: 正常
- 400: バリデーションエラー
- 404: 該当データなし
- 500: サーバエラー

## 5.2 共通レスポンス例
```json
{
  "data": {},
  "meta": {
    "requestId": "xxx",
    "version": "v1"
  },
  "errors": []
}
```

## 5.3 一覧系クエリ
- `keyword`
- `releaseId`
- `isTitleTrack`
- `memberId`
- `page`
- `size`
- `sort`

---

## 6. 非機能要件

- 一覧系 API は 1 秒以内の応答を目標
- 検索対象カラムと関連テーブルへインデックス付与
- `site_id` を含む複合インデックスを検討
- 将来の管理機能追加時に破綻しない命名と責務分離を行う
- DTO の後方互換追加を前提に項目設計する

## 7. テスト方針

### 単体テスト
- Service / UseCase
- 検索条件組み立て
- 別名検索ロジック
- データ変換ロジック

### 結合テスト
- Controller + DB
- 一覧取得
- 絞り込み取得
- 該当なし
- 不正 siteKey
- 不正 songId / memberId
- 複数作品紐付け時の代表作品取得
- 所属履歴から現在所属を組み立てる処理

### 完了条件
- 主要 API の正常系・異常系テストが通る
- 拡張用テーブルを経由しても返却順や表示内容が崩れない

---

## 8. 実装完了の定義
以下を満たしたらバックエンド完了とする。

- API 仕様書通りにレスポンスを返す
- 初期データで一覧 / 詳細が確認できる
- コール API で歌詞とコールの対応が崩れない
- `siteKey` を前提とした構成になっている
- `release_song`, `member_group_history`, 別名テーブル、マスタテーブルを含めた設計になっている
- `site` が将来のユーザー作成・公開制御に耐えられる設計になっている
- テストコードが揃っている
