---

## 1. 目的
Spring Boot で、=LOVE ファンサイト向けの参照系 API を実装する。  
ただし設計は将来的なマルチサイト展開を前提とし、`site` 起点でデータを扱えるようにする。

---

## 2. バックエンド実装スコープ

### 2.1 今回実装する API
- サイト情報取得
- 楽曲一覧取得
- 楽曲詳細取得
- 楽曲コール取得
- メンバー一覧取得
- メンバー詳細取得
- シングル一覧取得
- 検索条件用マスタ取得（必要に応じて）

### 2.2 今回実装しない API
- 管理画面向け CRUD
- 認証
- 画像アップロード
- サイト新規作成
- 利用者ごとのテーマ編集

---

## 3. 実装方針

## 3.1 パッケージ構成案
```text
com.example.fansite
├─ common
│  ├─ exception
│  ├─ response
│  ├─ config
│  └─ util
├─ domain
│  ├─ site
│  ├─ group
│  ├─ member
│  ├─ single
│  ├─ song
│  └─ call
├─ application
│  ├─ site
│  ├─ member
│  ├─ single
│  ├─ song
│  └─ call
├─ infrastructure
│  ├─ persistence
│  └─ external
└─ presentation
   └─ api
```

## 3.2 レイヤ方針
- `presentation`: Controller / request / response DTO
- `application`: UseCase / Service
- `domain`: Entity / domain service / repository interface
- `infrastructure`: JPA 実装、DBアクセス

## Step 3. 楽曲一覧 API 実装
### API
- `GET /api/v1/sites/{siteKey}/songs`

### 必須要件
- 楽曲名検索
- シングル絞り込み
- 表題曲フラグ絞り込み
- オリジナル歌唱メンバー取得
- 掲載シングル名取得
- ソート対応

### 完了条件
- 一覧画面が必要とする情報を 1 API で取得できる
- N+1 問題に配慮している

### 実装メモ
- 一覧はカード描画しやすい DTO に最適化する
- オリジナル歌唱メンバーは配列で返却
- 将来の多グループ対応を見越して `siteKey` 必須

---

## Step 4. 楽曲詳細 API 実装
### API
- `GET /api/v1/sites/{siteKey}/songs/{songId}`

### 必須要件
- 楽曲基本情報
- シングル情報
- 表題曲か否か
- オリジナル歌唱メンバー
- 説明文（任意）
- コール有無

### 完了条件
- 楽曲詳細ヘッダーに必要な情報が返る

---

## Step 5. 楽曲コール API 実装
### API
- `GET /api/v1/sites/{siteKey}/songs/{songId}/calls`

### 必須要件
- 歌詞ブロックごとに返せる
- 歌詞行とコール行の対応が取れる
- コール種別が識別できる
- 表示順を保証する

### 返却イメージ
```json
{
  "songId": 101,
  "blocks": [
    {
      "blockType": "VERSE",
      "orderNo": 1,
      "lines": [
        {
          "lineNo": 1,
          "lyrics": "君に会えたその日から",
          "calls": [
            {
              "callType": "CHANT",
              "text": "はい！"
            }
          ]
        }
      ]
    }
  ]
}
```

### 完了条件
- フロントが「歌詞の下にコール」を素直に表示できる

---

## Step 6. メンバー API 実装
### API
- `GET /api/v1/sites/{siteKey}/members`
- `GET /api/v1/sites/{siteKey}/members/{memberId}`

### 必須要件
- 一覧で最低限のプロフィールを返す
- 詳細で誕生日、メンバーカラー、出身地、ひとことを返す
- 卒業メンバー等を今後区別できる構造にする

### 完了条件
- メンバー一覧 / 詳細ページの表示に必要十分なデータが取れる

---

## Step 7. シングル API 実装

- シングル初期データ

### 完了条件

## 5. API 共通仕様

## 5.1 ステータスコード
- 200: 正常
- 400: バリデーションエラー
- 404: 該当データなし
- 500: サーバエラー

## 5.2 共通レスポンス例
```json
{
  "data": {},
  "meta": {
    "requestId": "xxx",
    "version": "v1"
  },
  "errors": []
}
```

## 5.3 一覧系クエリ
- `keyword`
- `singleId`
- `isTitleTrack`
- `memberId`
- `page`
- `size`
- `sort`

---

## 6. 非機能要件

- 一覧系 API は 1 秒以内の応答を目標
- 検索対象カラムへインデックス付与
- `site_key` を含む複合インデックスを検討
- 将来の管理機能追加時に破綻しない命名と責務分離を行う

---

## 7. テスト方針

### 単体テスト
- Service / UseCase
- 検索条件組み立て
- データ変換ロジック

### 結合テスト
- Controller + DB
- 一覧取得
- 絞り込み取得
- 該当なし
- 不正 siteKey
- 不正 songId / memberId

### 完了条件
- 主要 API の正常系・異常系テストが通る

---

## 8. 実装完了の定義
以下を満たしたらバックエンド完了とする。

- API 仕様書通りにレスポンスを返す
- 初期データで一覧 / 詳細が確認できる
- コール API で歌詞とコールの対応が崩れない
- siteKey を前提とした構成になっている
- テストコードが揃っている
