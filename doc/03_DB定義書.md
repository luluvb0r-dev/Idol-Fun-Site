## 1. 文書概要
本書は、=LOVE ファンサイトのバックエンドで使用する DB 定義書である。  
今回のサイトは =LOVE 固定だが、将来的なマルチサイト展開、複数グループ対応、所属履歴、検索別名、コール種別追加に加え、ユーザーが自分でファンサイトを作成・公開できるサービス形態を想定し、`site` を起点に拡張可能な構造とする。

---

## 2. 設計方針

### 2.1 基本方針
- 参照中心の読み取り最適化設計
- ただし将来の管理機能追加を見据えて正規化を維持
- 主要マスタは `site_id` を持たせる
- `site` は所有者と公開状態を持てるようにする
- 楽曲と作品は多対多
- 楽曲とメンバーは多対多
- メンバー所属は履歴テーブルで管理
- 検索別名は本体テーブルから分離
- コールは「ブロック > 行 > コール要素」の 3 階層で管理
- コール種別はマスタ参照で拡張可能にする

### 2.2 想定 DB
- PostgreSQL を第一候補
- MySQL でも実装可能

### 2.3 命名規則
- テーブル名: snake_case
- カラム名: snake_case
- 主キー: `id`
- 外部キー: `{table}_id`
- 論理削除は今回未採用
- 監査項目: `created_at`, `updated_at`

## 5. 変更判断の要点

- `song.single_release_id` は将来の多作品収録で破綻しやすいため廃止した
- `member.group_id` / `joined_on` / `generation_label` を本体から外し、履歴の正規データを `member_group_history` へ移した
- 別名列を増やさないため `song_alias`, `member_alias` を追加した
- コール種別増加に備えて `call_type_master` を追加した
- ユーザー作成サイトに備えて `site` に所有者、公開状態、初期サイト区分を追加した

---

## 6. 初期データ方針

- `site` に `equal-love` を登録
- `owner_user_id` は null、`is_seeded_site` は true として登録
- `idol_group` に =LOVE を登録
- `site_theme_setting` にピンク系テーマを登録
- `call_type_master` に `CHANT`, `CLAP`, `MIX`, `JUMP`, `OTHER` を登録
- メンバー / 所属履歴 / 作品 / 楽曲 / 別名 / コールを初期投入する
- 初期データは SQL または migration seed で管理する

---

## 7. 将来拡張の考え方

今回の要件外だが、将来以下へ拡張しやすい構造とする。

- サイト単位で別アイドルを追加
- グループごとのメンバー履歴管理
- 複数バージョンのコール管理
- 楽曲ごとのメディアリンク
- ユーザー投稿型ファンサイト

そのため、今回から `release_song`, `member_group_history`, 別名テーブル、コール種別マスタに加え、`site` の所有者・公開状態管理を採用している。

## 8. 補足
著作権や利用許諾の問題がある場合、歌詞データは表示用の管理対象テキストのみ保持する運用も検討すること。  
また、検索要件が増えた場合は `normalized_text` に対して全文検索や N-gram の導入を別途検討する。
