## 1. 目的
Next.js ベースのフロントエンドで、=LOVE ファンサイトを **モダンかつピンク基調** のデザインで実装する。  
検索しやすさ、楽曲情報の見やすさ、コールの判別しやすさを重視しつつ、将来の機能追加で UI の作り直しが最小になる構成にする。

---

- Card Based UI

## 2.2 カラートーン案
- Primary: やわらかいピンク

### 表示項目

## Step 6. 楽曲詳細ページ実装
### ヘッダー表示
- 楽曲名
- 代表作品名
- 複数作品がある場合の作品一覧
- 表題曲か否か
- オリジナル歌唱メンバー

### コール表示
- 歌詞行
- その直下にコール表示
- コール種別バッジ
- 掛け声とクラップの色分け
- 長文でも読みやすい縦レイアウト

### UX 要件
- 歌詞とコールの対応が崩れない
- コールは補助的だが視認性が高い
- スクロールしても見失いにくい
- モバイルでも行間を確保
- コール種別追加時も既存コンポーネントで表示できる

### 完了条件
- 楽曲詳細で「見ながらコール確認」がしやすい

---

## Step 9. 状態設計
### 必須状態
- Loading
- Error
- Empty
- Success

### 実装方針
- API 呼び出し中は Skeleton を優先
- エラー時は再試行導線を出す
- 0件時は説明文とフィルタ解除導線を出す
- 将来の項目追加に備えて null-safe な表示を徹底する

## 6. 型定義方針
API レスポンスの型をフロント側でも厳密に管理すること。

- Site
- SongSummary
- SongDetail
- ReleaseSummary
- SongCallBlock
- SongCallLine
- SongCallItem
- CallTypeMeta
- MemberSummary
- MemberDetail
- GroupHistoryItem

### 補足
- API 生レスポンス型と画面描画型を分ける
- `callTypeCode` を UI で直接分岐しすぎず、表示メタデータを優先利用する

---

## 7. アクセシビリティ要件

- コール色分けは色覚差を考慮する
- 色だけでなくラベルを表示する
- 文字サイズは最小でも可読性を担保する
- ボタンやリンクのタップ領域を十分に確保する
- コントラスト比に配慮する

---

## 8. 完了判定
以下を満たしたらフロント実装完了とする。

- トップ / 楽曲一覧 / 楽曲詳細 / メンバー一覧 / メンバー詳細が動作する
- 楽曲一覧で検索と絞り込みができる
- 楽曲詳細で歌詞の下にコールが表示される
- 掛け声とクラップが色分けされる
- 代表作品と複数作品の両方を扱える
- メンバー詳細で所属履歴を扱える構造になっている
- モバイル表示でも崩れない
- Loading / Error / Empty に対応している
