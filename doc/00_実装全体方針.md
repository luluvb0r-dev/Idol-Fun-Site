## 1. 文書概要
本書は、Antigravity に対して **=LOVE のファンサイト** を実装依頼するための全体方針書です。  
今回のスコープは以下の 3 機能を最優先とします。

1. 楽曲一覧
2. 楽曲ごとのコール表示
3. 簡易メンバープロフィール表示

今回は =LOVE 固定のサイトとして実装する一方で、**将来的に任意のユーザーが独自のアイドルファンサイトを作成できる構成**を見据え、DB 設計・API 設計・画面設計をサイト単位で拡張できるようにします。  
また、今回は =LOVE の初期サイトを先に実装するが、後日ユーザー自身が内容を追加・変更できるサービス形態へ拡張することを前提とします。

---

## 2. 前提条件

- フロントエンドは **Next.js ベース**
- バックエンド API は **Spring Boot**
- 既に最低限のプロジェクトは存在するため、**環境構築は実施不要**
- 本依頼では、**実装方針・画面方針・API設計・DB設計・ER図・仕様書整備**を含める
- 仕様書は必ず **改訂履歴を先頭に持つ**
- バージョニングを徹底する

---

## 3. 今回のスコープ

## 3.1 必須機能

### 3.1.1 楽曲一覧
以下を一覧表示できること。

- 楽曲名
- ふりがな / 検索補助項目
- オリジナル歌唱メンバー
- 代表掲載作品
- 表題曲か否か
- 楽曲詳細画面への導線

### 3.1.2 楽曲検索
楽曲一覧画面で以下の検索ができること。

- 楽曲名の部分一致検索
- ふりがな / ローマ字 / 別名検索に将来拡張しやすい設計
- 作品絞り込み
- 表題曲 / カップリングの絞り込み
- メンバー名による絞り込み

### 3.1.3 コール表示
楽曲詳細画面で以下を表示すること。

- 歌詞
- 各歌詞の下に対応するコール
- コール種別を視認しやすく色分け
- 掛け声、クラップ以外の種別追加に耐えられる構造

### 3.1.4 メンバープロフィール
表示項目は最小限とし、情報量を増やしすぎないこと。

- 名前
- 誕生日
- メンバーカラー
- 出身地（任意）
- 加入期 / 加入日（任意）
- ひとこと紹介（任意・短文）

なお、ユーザー認証・サイト編集機能は今回は実装しないが、**サイト所有者情報と公開状態を後付けしやすい構造**にしておくこと。

## 5. 拡張性を最優先にした設計原則

### 5.1 今回の設計で必ず守ること

- 画面都合の一時的な見せ方を、永続データ構造に直結させない
- 一対一に見える関係でも、将来の多対多化可能性を先に評価する
- `enum` の追加頻度が高い要素は、コード値固定ではなくマスタ化を優先する
- 現在値だけでなく、履歴・別名・並び順の変化を保持できる構造にする
- API は「今の画面に必要な最小項目」だけでなく、「将来の画面追加時に再利用しやすいまとまり」で返す

### 5.2 変更すべき重要ポイント

#### 1. 楽曲と作品を直接 1 対多で結びすぎない
現在の要件では楽曲に掲載作品が 1 件あれば足りますが、将来は以下が起こり得ます。

- 同一楽曲が複数シングル・アルバム・配信作品に収録される
- 代表作品と再録作品を区別したい
- 盤種ごとの差分や順序を持ちたい

そのため、`song.single_release_id` の直結ではなく、**`release_song` の中間テーブルで結ぶ設計**を基本とします。

#### 2. メンバー所属を現在値だけで持たない
将来的に以下が起こり得ます。

- グループ兼任
- 卒業 / 再加入
- 期や役割の変更

そのため、`member.group_id` のみで管理せず、**`member_group_history` を持って所属履歴を正規化**します。

#### 3. 検索性向上のため、別名を本体テーブルへ埋め込まない
検索拡張でありがちな失敗は、`title_kana`, `title_roman`, `alias1`, `alias2` のように列を増やし続けることです。  
これを避けるため、**`song_alias` と `member_alias` を分離**し、別名種別ごとに増やせる構造にします。

#### 4. コール種別はマスタ化する
`CHANT`, `CLAP`, `MIX`, `JUMP` だけでは将来的に足りなくなる可能性が高いです。  
そのため、`song_call_item.call_type` を固定列挙として扱うのではなく、**`call_type_master` を参照する構造**を前提にします。

#### 5. 表示用の色やラベルはデータとして返せるようにする
フロントが色分けロジックをハードコードすると、サイトごとの演出変更やコール種別追加時に改修範囲が広がります。  
そのため、**API は種別コードだけでなく、表示ラベル・色・アイコンキーなども返却可能な構造**にします。

#### 6. サイトの所有者と公開状態を初期設計に含める
ユーザーが自由にファンサイトを作るサービスでは、少なくとも以下が必要になります。

- 誰がそのサイトを作成したか
- 公開前の下書きサイトか、公開中サイトか
- 運営が用意した初期サイトか、ユーザー作成サイトか

今回は認証や編集 UI を実装しないが、**`site` に所有者参照用の列と公開状態を持てる構造**を用意しておく。

## 6. アーキテクチャ方針

## 6.1 全体構成
- フロントエンド: Next.js ベースの Web アプリ
- バックエンド: Spring Boot REST API
- DB: RDBMS（PostgreSQL を第一候補、MySQL も選択可能）
- 表示対象サイト: 当面は `=LOVE` 固定

## 6.2 将来拡張方針
将来的に以下が可能なように設計すること。

- サイトごとに別アイドル / 別グループを管理
- 同一楽曲を複数作品へ紐づけ
- 同一メンバーの所属履歴を保持
- 1つの API 基盤で複数のファンサイトを提供
- サイトごとにテーマやブランド設定を切り替え
- 検索用別名や表示ルールをサイトごとに拡張
- ユーザーごとにサイトを作成し、後から編集できるようにする

このため、主要データは `site` を起点に紐づけ、履歴・別名・マスタを後付けではなく初期設計から含めます。

## 7. 実装順の推奨

以下の順番で実装を進めることを推奨します。

### Step 1. バックエンド基盤と DB 実装
理由:

- フロントエンドの表示要件が API 仕様に強く依存するため
- 将来拡張を見越したデータ構造を先に固めるべきため
- 履歴管理や多対多設計は後から差し替えるコストが高いため

実施内容:

- DB スキーマ作成
- マスタ / 履歴 / 別名テーブルを含む基盤設計
- Entity / Repository / Service / Controller 実装
- 楽曲一覧 API
- 楽曲詳細 API
- 楽曲コール API
- メンバー一覧 / 詳細 API
- 作品一覧 API
- 初期データ投入方式の整備

### Step 2. フロントエンド実装
実施内容:

- 共通レイアウト
- トップページ
- 楽曲一覧画面
- 楽曲詳細画面
- メンバー一覧画面
- メンバー詳細画面
- 検索 UI
- コール色分け UI
- レスポンシブ対応

## 8. デザイン方針

## 8.1 トーン&マナー
- **ピンク基調**
- モダン
- 清潔感
- 可読性重視
- スマホで見やすい
- 初見でも使いやすい UX

## 8.2 デザインキーワード
- Soft Pink
- White Space
- Rounded UI
- Card Layout
- Sticky Search
- Fast Navigation
- Minimal Noise

## 8.3 UI 原則
- 主要操作は 2 タップ以内で到達できること
- 一覧画面では検索と絞り込みを上部固定
- 情報の優先度を明確にする
- コールは歌詞よりやや補助的に見せるが、判別はしやすくする
- 色だけでなくラベル・アイコンでも区別できるようにする

---

## 9. データ設計の重要方針

以下を必須とすること。

- 楽曲と作品は中間テーブルで関連付ける
- 楽曲と歌唱メンバーは多対多で管理する
- メンバーの所属情報は履歴テーブルで管理できるようにする
- 楽曲名やメンバー名の検索用別名は別テーブルで管理する
- コールは「歌詞行」と「コール行」の対応が取れる構造にする
- コール種別はマスタ参照で追加可能にする
- =LOVE 固定実装であっても、`site` を起点に拡張できる構造にする
- サイトには所有者情報、公開状態、運営初期データかどうかを持てるようにする

## 11. 品質要件

- API は RESTful を基本とする
- 一覧 API はソート・検索・絞り込みに対応する
- フロントはローディング / エラー / 空状態を必ず実装する
- コールの色分けはアクセシビリティを考慮する
- 型定義を明確にする
- 仕様書と実装の差異が出ないようにする
- 将来の拡張時にテーブル再設計を最小化する
- マスタ追加や履歴追加で API の互換性を壊しにくい設計にする

---

## 12. 受け入れ条件

- 楽曲一覧画面で検索できる
- 楽曲一覧画面で以下が確認できる
  - オリジナル歌唱メンバー
  - 代表掲載作品
  - 表題曲か否か
- 楽曲詳細画面で歌詞の直下にコールが表示される
- 掛け声とクラップが色分けされている
- メンバー一覧 / 詳細で最低限のプロフィールが確認できる
- DB 定義書 / ER 図 / API 仕様書が揃っている
- API とフロントでサイト単位拡張を見据えた構造になっている
- 多作品収録、所属履歴、別名検索、コール種別追加に設計上対応できる
- 将来のユーザー作成サイト追加時に `site` の再設計が不要である

---

## 13. 実装時の注意点

- 今回は著作権配慮のため、歌詞データは本番公開前に権利整理が必要になる可能性がある
- 開発時はダミーデータまたは管理対象データで進行できる構成にする
- コールデータは行単位で柔軟に差し替えられるようにする
- =LOVE 専用実装に寄りすぎず、サイトキーで切り替え可能な構成にする
- 画面都合の簡略項目は API DTO 側で吸収し、DB には将来の履歴や多対多を耐えられる形で保持する
