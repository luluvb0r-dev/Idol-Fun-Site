# =LOVEファンサイト 実装全体方針書

## 1. 文書概要
本書は、Antigravity に対して **=LOVE のファンサイト** を実装依頼するための全体方針書です。  
今回のスコープは以下の 3 機能を最優先とします。

1. 楽曲一覧
2. 楽曲ごとのコール表示
3. 簡易メンバープロフィール表示

なお、今回は =LOVE 固定のサイトとして実装する一方で、**将来的に任意のユーザーが独自のアイドルファンサイトを作成できる構成**を見据えて、DB設計・API設計はサイト単位で拡張できるようにします。

---

## 2. 前提条件

- フロントエンドは **Node.js ベース**
- バックエンド API は **Spring Boot**
- 既に最低限のプロジェクトは存在するため、**環境構築は実施不要**
- 本依頼では、**実装方針・画面方針・API設計・DB設計・ER図・仕様書整備**を含める
- 仕様書は必ず **改訂履歴を先頭に持つ**
- バージョニングを徹底する

---

## 3. 今回のスコープ

## 3.1 必須機能

### 3.1.1 楽曲一覧
以下を一覧表示できること。

- 楽曲名
- ふりがな / 検索補助項目（検索性向上のため）
- オリジナル歌唱メンバー
- 掲載シングル
- 表題曲か否か
- 楽曲詳細画面への導線

### 3.1.2 楽曲検索
楽曲一覧画面で以下の検索ができること。

- 楽曲名の部分一致検索
- ふりがな / ローマ字 / 別名検索に将来拡張しやすい設計
- シングル絞り込み
- 表題曲 / カップリングの絞り込み
- メンバー名による絞り込み（将来拡張しやすい設計）

### 3.1.3 コール表示
楽曲詳細画面で以下を表示すること。

- 歌詞
- 各歌詞の下に対応するコール
- コール種別を視認しやすく色分け
  - 掛け声
  - クラップ
  - 任意拡張（MIX / 合いの手 / ジャンプ等）に対応可能な設計

### 3.1.4 メンバープロフィール
表示項目は最小限とし、情報量を増やしすぎないこと。

- 名前
- 誕生日
- メンバーカラー
- 出身地（任意）
- 加入期 / 加入日（任意）
- ひとこと紹介（任意・短文）

- コメント機能

ただし、将来の拡張を見据えて、**DB と API はマルチサイト対応を想定した設計**にすること。

---

## 5. アーキテクチャ方針

## 5.1 全体構成
- フロントエンド: Node.js ベースの Web アプリ
- バックエンド: Spring Boot REST API
- DB: RDBMS（MySQL または PostgreSQL を想定）
- 表示対象サイト: 当面は `=LOVE` 固定

## 5.2 将来拡張方針
将来的に以下が可能なように設計すること。

- サイトごとに別アイドル / 別グループを管理
- 同一楽曲構造・コール構造を再利用
- 1つの API 基盤で複数のファンサイトを提供
- サイトごとにテーマカラーやブランド設定を切り替え

このため、主要データは `site` を起点に紐づくよう設計すること。

## 9. データ設計の重要方針

以下を必須とすること。

- 楽曲とシングルを分離して管理する
- 楽曲と歌唱メンバーは多対多で管理する
- コールは「歌詞行」と「コール行」の対応が取れる構造にする
- コール種別は enum 固定にしすぎず、将来追加可能な構造にする
- =LOVE 固定実装であっても、`site` または `tenant` を起点に拡張できる構造にする

## 11. 品質要件

- API は RESTful を基本とする
- 一覧 API はソート・検索・絞り込みに対応する
- フロントはローディング / エラー / 空状態を必ず実装する
- コールの色分けはアクセシビリティを考慮する
- 型定義を明確にする
- 仕様書と実装の差異が出ないようにする
- 将来の拡張時にテーブル再設計を最小化する

---

## 12. 受け入れ条件

- 楽曲一覧画面で検索できる
- 楽曲一覧画面で以下が確認できる
  - オリジナル歌唱メンバー
  - 掲載シングル
  - 表題曲か否か
- 楽曲詳細画面で歌詞の直下にコールが表示される
- 掛け声とクラップが色分けされている
- メンバー一覧 / 詳細で最低限のプロフィールが確認できる
- DB定義書 / ER図 / API仕様書が揃っている
- API とフロントでサイト単位拡張を見据えた構造になっている

---

## 13. 実装時の注意点

- 今回は著作権配慮のため、歌詞データは本番公開前に権利整理が必要になる可能性がある
- 開発時はダミーデータまたは管理対象データで進行できる構成にする
- コールデータは行単位で柔軟に差し替えられるようにする
- =LOVE 専用実装に寄りすぎず、サイトキーで切り替え可能な構成にする
